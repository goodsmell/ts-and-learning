# 문제 #8

## 막혔던 부분

```
type PowerUser = Omit<User & Admin, 'type'> & { type: 'powerUser' };
```

이렇게 했더니, roler과 occupation에 `TS2322: Type 'string' is not assignable to type 'never'.`라는 에러가 발생했다.

찾아보니, User의 type은 'user'이고, Admin의 type은 'admin'으로 서로 다른 리터럴 타입을 가지고 있는데, 이 둘을 &연산으로 합쳐버리면서 생기는 문제였다.

User&Admin을 해버리면 `type: 'user'&'admin`이 되는 꼴이다. 현실적으로 type이 user과 admin 둘 다 인 경우는 없기 때문에 `type:never`을 해버리는 것이다. TypeScript는 이렇게 물리적으로 불가능한 상태를 감지하면 해당 타입을 never(공집합)로 처리해버린다고 한다.

근데 Omit으로 문제가 되는 type을 지워버렸는데 왜 문제일까?
논리적으로는 type을 지우고 새로 붙였으니 문제가 없어 보이지만, TypeScript 컴파일러는 User & Admin을 결합하는 과정에서 구조적 불일치를 먼저 판단한다고 한다.
특히 User과 Admin은 type으로 식별 가능한 유니온이기 때문에 이를 강제로 합치려 하면 타입스크립트는 혼란에 빠진다.

근데 왜 에러는 뜬금없이 role과 occupation에 생겼을까?
PowerUser타입 자체가 never가 되어버렸기 때문이다.
User & Admin을 계산 할 때 식별자가 다른 두 객체 타입을 합치려고 하니 논리적으로 존재할 수 없는 객체라고 판단하고 User & Admin 전체를 `never`라고 결론을 내리게 되는 것이다. 즉 객체 자체가 성립 불가하다고 판정받은 것.

1. User & Admin 계산 => never 판정
2. Omit<never, 'type'> 계산 => never은 아무것도 없는 상태이니 거기서 type을 빼도 never
3. never & { type: 'powerUser' } 계산 => 존재할 수 없는 것 (never)과 어떤 것을 합쳐봤자 never

근데 왜 객체 전체가 아니라 role과 occupation에만?
Person 유니온의 축소 때문 `Person = User | Admin | PowerUser` 이었는데 PoserUser가 never이니까 `Person = User |Admin`인 상태이다. 따라서 공통 속성인 name, age만 통과되는 것.

## 해결을 위해 필요한 키워드 1개

Intersection Type

## 키워드 단순 학습

교집합(Intersection)
A이기도 하면서 동시에 B여야 한다.
여러 타입을 융합하여 만든 하나의 타입. "&" 연산자를 활용하여 정의한다.
융합된 타입들의 요소를 전부 가지고있다.

## 문제에 적용시켜 해결한 방법

Omit으로 먼저 각 객체에서 type을 제거하고 &로 묶자!
{type : 'powerUser'}도 &로 묶자!

## 키워드 심화학습 (생각해 볼 만한 질문)

매번 Omit을 여러 개 중첩해서 사용하는 방식이 코드의 가독성이나 유지보수 측면에서 최선일까

## 자바스크립트 였다면 어떤 문제가 생겼을까?

## 한 줄 요약

식별 가능한 속성이 충돌하는 타입들을 합칠 때는, 반드시 충돌 지점을 먼저 제거(Omit)해야 never의 늪에 빠지지 않는다.
