# 문제 #9

## 막혔던 부분

requestCoffeeMachineQueueLength 함수에서 response 데이터 타입이 ApiResponse<`number`>인지 모르겠다.
이유를 찾아보니 지금 함수는 에러를 뱉는 역할만 하고 있지만, requestCoffeeMachineQueueLength라는 함수 이름과, 성공했을 때의 데이터 타입을 생각해 봤을 때 number라는 것이다.

```
// 성공 시나리오 (미래의 모습)
callback({
    status: 'success',
    data: 5 // 대기열 숫자는 'number' 타입이어야 하죠!
});
```

대기열의 길이는 논리적으로 숫자이기 때문에 에러가 나더라도 이 함수의 본질적인 응답 규격은 number를 담는 그릇이어야 한다.

## 해결을 위해 필요한 키워드 1개

제너릭

## 키워드 단순 학습

타입을 직접적으로 고정된 값으로 명시하지 말고 변수를 통해 언제든지 변할 수 있는 타입을 통해 유연하게 코딩할 수 있는 장치.

## 문제에 적용시켜 해결한 방법

ApiResponse<T>라는 제네릭 타입을 통해, 성공(success) 시 반환할 data의 타입을 외부에서 주입받도록 설계했다.

## 키워드 심화학습 (생각해 볼 만한 질문)

질문 1: 만약 ApiResponse<number> 대신 ApiResponse<any>를 쓴다면 어떤 타입 안정성 문제가 발생할까?

질문 2: 에러 발생 시에도 data 필드가 강제된다면(예: {status: 'error', data: T, error: string}), 이때 data에는 어떤 값을 넣어야 할까? 제네릭 설계 시 '성공'과 '실패'의 타입을 어떻게 분리하는 게 좋을까?

## 자바스크립트 였다면 어떤 문제가 생겼을까?

JS에서는 이 함수가 숫자를 주는지, 문자열을 주는지, 혹은 에러 객체만 주는지 코드를 실행해 보기 전까지 알 수 없다.
startTheApp에서 response.data를 사용할 때, 그것이 숫자라고 확신하고 Math.round() 같은 함수를 썼다가 undefined가 들어와 프로그램이 터지는 상황을 방지하기 어렵다.

## 한 줄 요약

제네릭은 현재의 결과뿐만 아니라, 함수가 약속한 미래의 데이터 규격까지 안전하게 정의하는 도구이다.
