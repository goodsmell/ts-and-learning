# 문제 #4

## 막혔던 부분

이 문제가 왜 틀린 문제인지조차 알 수 없었다.

isAdmin 함수를 쓰지 않고 if문에 타입을 비교하는 조건을 넣어주면 통과가 되는 문제이다.

왜 사용자가 직접 if (person.type === 'admin')이라고 쓰면 성공하고, isAdmin(person) 함수를 호출하면 TypeScript가 타입을 좁혀주지 못할까?

## 해결을 위해 필요한 키워드 1개

사용자 정의 타입 가드

## 키워드 단순 학습

타입 가드는 조건문 안에서 타입을 좁혀주는 방법
타입 가드 방법

1. typeof
2. instanceof
3. in
4. 리터럴 타입 가드
5. 사용자 정의 타입 가드

### 사용자 정의 타입 가드란 ?

타입을 좁혀주는 함수를 만드는 것.
isTypeName 형태의 함수명을 자주 사용하며, is 연산자를 사용하여 반환 타입을 정의하는 행위.
리턴값은 boolean.

### is 연산자?

is 연산자는 함수의 결과값과 입력 인자의 타입 사이의 논리적 연결 고리를 만드는 장치이다.

## 문제에 적용시켜 해결한 방법
지금 문제에 있는 함수는 그냥 일반 함수이다. 
사용자 정의 타입 가드 함수로 사용하려면 `parameterName is Type`을 붙여줘야 한다.

## 키워드 심화학습 (생각해 볼 만한 질문)

#### 질문 1: isAdmin 함수가 리턴하는 person.type === 'admin'의 결과값은 타입은?

답 : true

#### 질문 2: 함수 밖에서 그 결과를 받았을 때, TypeScript가 "아, 이 함수가 true를 뱉었으니 매개변수로 넣었던 person은 무조건 Admin이겠구나!"라고 추론할 근거가 현재 코드에 있을까?

답: 직접 비교할 때에는 TypeScript가 현재 스코프 내에서 직접 type 리터럴을 확인할 수 있어서 타입을 즉시 좁힐 수 있다.
그러나 함수로 분리하여 비교하면 타입스크립트는 해당 함수의 정의 (시그니처)만 본다. 지금 isAdmin의 정의(시그니처)는 `(person: Person) => boolean`. 이 정의만 봐서는 true가 나온다고 해서 어떤 타입인지 추론할 수 없다.

### 결론

타입 스크립트는 함수를 거쳐 나오는 true/false만으로는 그게 타입을 좁혀야하는 근거로 받아들이지 못한다. 그래서 `parameterName is Type`라는 특수한 반환 타입을 사용해 그 연결 고리를 만들어 줘야 한다.

심화 키워드 : Narrowing
