### 1. 첫 학습 전략 설계

- 어떤 전략으로 학습하기로 했는가?

    일단 풀어보고, 이 문제에서 내가 얻을 수 있는게 무엇인지 알아본다.

- 그 전략을 선택한 이유는?

    어떤 것을 알아야 하고 중요한 것이 무엇인지 인지하기 위해서이다.

- 실제로 어떻게 학습했는지 디테일한 과정을 써보세요.

    ```
    1. 문제를 푼다.
    2. < 성공했을 경우 > 해당 문제에서 집중해야할 키워드가 무엇인지 생각한다.
    2. <못 풀었을 경우>키워드에 대한 힌트를 ai에게 얻는다.
    3. 해당 키워드를 책이나 ai를 통해 학습한다.
    4. 다른 방식으로 풀 수 있는지 확인한다.
    ```

### 2. 전략 평가

- 효과적이었던 것과 그 이유

- 비효과적이었던 것과 그 이유

    너무 해당 키워드에 대해 깊게 파고들려고 한다.

    아는 내용에 대해서도 더 제대로 알고가려고 해서 시간이 너무 걸린다..

### 3. AI 피드백

> 너무 1부터 다 알려고 하는 습관을 바꾸기 위해 파인만 학습법을 역으로 사용해보라.
>
> "이 개념을 중학생에게 3분 안에 설명할 수 있는가?"를 기준으로 삼고, 만약 설명 가능하다면 그 이상의 탐구는 멈추고 다음 탐구로 넘어가라.
>
> 프로젝트를 하다가 막히면 그 때 더 깊은 탐구를 이어가라

> 이 개념을 내 프로젝트 어디에 적용해 볼 수 있을까? 라는 문장을 생각해봐라

> 현재 풀고 있는 문제와 직접 관련된 개념까지만 학습해라.

### 4. 다음 타임에 바꿀 것

- 바꿀 것과 그 이유
    - 해당 키워드를 책이나 ai를 통해 학습할 때 현재 풀고있는 문제와 직접 관련된 개념까지만 학습하도록 하고, 중학생에게 설명할 수 있을 정도의 이해만 한다.
    - 그리고 각 문제에서 내 프로젝트 어디에 이런 것을 적용해 볼 수 있을까? 생각해본다.
    - 막히는 문제는 '이현'이 공유해준 템플릿을 적용해본다.

```
문제

문제에 적용시켜 해결한 방법
```

## 학습 로그 #2

**시간**: 03/25 10:40 ~ 16:30 (약 120분)

**학습 범위**: typescript-exercises #4~#6

### 1. 이번 타임의 학습 전략

- 이전에 바꾸기로 한 전략은 무엇이었고, 실행했는가?
    - 해당 키워드를 책이나 ai를 통해 학습할 때 현재 풀고있는 문제와 직접 관련된 개념까지만 학습하도록 하고 하였다.
    - 문제마다 템플릿대로 생각을 정리했다.
    - 그리고 각 문제에서 내 프로젝트 어디에 이런 것을 적용해 볼 수 있을까? 생각해보기로 햇던 전략은 학습 도중 제거하였다.

- 실제로 어떻게 학습했는지 디테일한 과정을 써보세요.

```
1. 문제 풀기

< 안풀어질 경우 >

2. 책 목차를 보며, 해당 문제를 풀기 위해 필요할 것 같은 키워드 찾아보기
3. 키워드가 떠오르지 않으면 ai에게 키워드 묻기.
4. 키워드만 공부하기
5. 문제 재시도

< 풀어질 경우 >

6. 각 문제 md파일에 내용 정리.
```

### 2. 전략 평가

- 효과적이었던 것과 그 이유

    각 문제 별로 md파일을 작성한 것이 효과적이 었던 것 같다.
    문제를 푸는 것에서 넘어가는 것이 아니라 그 문제에서 핵심이 무엇이고, 주요 개념이 무엇이고, 어떻게 풀어냈는지 정리가 되기 때문이다.

- 비효과적이었던 것과 그 이유

    각 문제에서 내 프로젝트 어디에 이런 것을 적용해 볼 수 있을까? 생각해보는 것이 이번 전략에 포함되어 있었는데, 아직 개념이 제대로 잡히지 않아 여기까지 생각하기에는 무리가 있었던 것 같다.

### 3. AI 피드백

- 자신의 학습 전략에 대해 AI에게 피드백을 요청하고, 유용했던 제안 1가지 이상 기록
    1. ai에게 내가 찾은 키워드 외에 더 효율적인 typeScript 기능이 있는가? 물어보며 피드백의 밀도 높히기.
    2. md 파일에 '나중에 이 개념이 헷갈릴 때 다시 볼 나만의 한 줄 요약'을 적어보며 뇌의 인출 속도를 높혀보자
    3. 이 문법이 없었다면 javascript에서 어떤 런타임 에러가 났을까? 상상해보기.
    4. md파일을 동료에게 설명하듯 구어체로 써보자!

### 4. 다음 타임에 바꿀 것

- 유지할 것과 그 이유

    추론형 키워드 도출과 문제별 기록하는 방식을 유지할 것이다.

    문제가 막혔을 때 바로 답을 보지 않고 목차 → AI 키워드 질문 → 개념 공부 순으로 접근하는 방식이 머릿속에 더 각인시킬 수 있는 것 같다. 또한 .md 파일에 기록하는 작업을 통해 파편화된 정보를 하나로 묶어지는 것 같다.

- 바꿀 것과 그 이유
    - 프로젝트 적용 고민을 해보기 -> 에러 시나리오 상상해보기로 변경할 것이다.

    아직 개념이 생소한 상태에서 거창한 프로젝트 적용을 고민하는 것은 외재적 인지 부하만 높일 뿐이라는 ai 피드백을 받았다. 대신 '만약 이 타입이 없어서 순수 JS로 짰다면, 어떤 에러가 생겼을까?' 를 한 문장으로 적어보는 것이 더 효과적이다.
    - md 파일에 '나중에 이 개념이 헷갈릴 때 다시 볼 나만의 한 줄 요약'을 적어보며 뇌의 인출 속도를 높혀볼 것이다.

## 학습 로그 #3

**시간**: 03/26 10:30 ~ 12:00 (약 90분)

**학습 범위**: typescript-exercises #7~#8

### 1. 이번 타임의 학습 전략

- 이전에 바꾸기로 한 전략은 무엇이었고, 실행했는가?
    - 각 문제 md 파일에 나중에 이 개념이 헷갈릴 때 다시 볼 나만의 한 줄 요약을 적어보기로 했는데, 실행했다.
    - js 였을 경우 에러 시나리오 상상해보기로 했는데 그냥 짧게 상상만 하고 구체화하진 않았다..

- 실제로 어떻게 학습했는지 디테일한 과정을 써보세요.

```
1. 문제 이해하기
2. 타입스크립트 문서를 보며 해당 문제를 푸는 데 필요할 것 같은 키워드 확인하기
3. 문제에 적용하기
4. 해결 된다면, 그 키워드에 대해 공부하기
5. 해결되지 않는다면 ai에게 힌트 제공 받기
6. 문제 md파일에 생각 과정을 정리하며 에러 시나리오 상상
```

### 2. 전략 평가

- 효과적이었던 것과 그 이유

    이 전과 마찬가지로 문제 마다 md 파일을 만든것이 가장 학습에 효과적이었다. 내가 어떤 것에서 막혔고 헷갈려 했고, 어떻게 해결했는지 생각만 하는게 아니라 글로 적으니 정리가 되는 것 같다. 또한 js였다면 어땠을 지 짧게라도 생각해 보는 것이 ts와 js의 차이를 알게되어 좋았던 것 같다.

- 비효과적이었던 것과 그 이유

    책으로 키워드 찾기.

    원래는 책으로 키워드를 찾았는데, 그러다 보니 시간이 오래 걸리고 선택한 책에 해당 개념이 상세히 나와있지 않은 경우도 있었다. 그래서 타입스크립트 공식 문서를 보며 키워드를 고르는 전략으로 변경했다.

### 3. AI 피드백

- 자신의 학습 전략에 대해 AI에게 피드백을 요청하고, 유용했던 제안 1가지 이상 기록
    - 그 한 줄 요약이 정말 '나의 언어'인지 점검해 보세요. 만약 "제네릭은 타입을 변수처럼 사용하는 것이다"라고 적었다면, "제네릭은 타입 구멍을 뚫어놓고 나중에 채워 넣는 마법이다"처럼 본인이 가장 직관적으로 이해할 수 있는 비유를 섞어보세요.

### 4. 다음 타임에 바꿀 것

- 유지할 것과 그 이유

    전체적인 방식은 유지한다. 책이 아닌 공식문서에서 키워드를 찾는 방식을 유지한다. 원하는 정보를 더 방대하고 빠르게 찾아낼 수 있어 학습 효율이 높은 것 같기 때문이다.

- 바꿀 것과 그 이유

    한 줄 요약 작성시, 나만의 언어로 머릿속에 들어올 수 있도록 작성한다.
    단순히 텍스트를 옮겨 적는 것이 아니라, 완전히 내 것으로 소화하여 실무나 협업 시 즉시 인출할 수 있도록 하기 위함이다.

    한 줄 요약 작성시, 나만의 언어로 머릿속에 들어올 수 있도록
    `제네릭은 현재의 결과뿐만 아니라, 함수가 약속한 미래의 데이터 규격까지 안전하게 정의하는 도구이다.` 라고 작성해보았다.

- 실제로 어떻게 학습했는지 디테일한 과정을 써보세요.

```
1. 문제 이해하기
2. 타입스크립트 문서를 보며 해당 문제를 푸는 데 필요할 것 같은 키워드 확인하기

4. 해결 된다면, 그 키워드에 대해 공부하기
5. 해결되지 않는다면 ai에게 힌트 제공 받기
6. 내 답과 정답이 왜 다른지, 내 답으로 진행할 경우 어떤 문제가 생기는지 생각해보기.
7. 문제 md파일에 생각 과정을 작성하며 에러 시나리오 상상 (한줄 요약 작성)
```

### 2. 전략 평가

- 효과적이었던 것과 그 이유

    ai에게 내 답 피드백 받기

    내 답이 테스트 통과는 되지만 정답과는 다른 경우가 빈번하게 발생했다. 내 답과 정답이 다른 이유를 추측하고, 이해한 내용을 ai에게 물어보면서 학습하였더니, 개념 정리도 되면서 옳은 것과 잘못된 것을 구별하는 능력이 생기는 것 같다.

- 비효과적이었던 것과 그 이유

    해당 키워드에 대해 짧게 공부하는 것.
    비효과적이라기 보단 output에 집중하려다 보니 너무 얕게 공부하는게 아닌가? 싶은 의구심이 든다..

### 3. AI 피드백

- 자신의 학습 전략에 대해 AI에게 피드백을 요청하고, 유용했던 제안 1가지 이상 기록

    > AI에게 단순히 맞는지 묻지 말고, "성능, 가독성, 유지보수 측면에서 정답이 내 코드보다 우월한 이유가 뭐야?"라고 압박해 보자.

    > 지금은 모든 뿌리를 다 캐낼 때가 아니라, 일단 '청킹(Chunking)'을 만드는 시기입니다. 얕게 여러 번 반복해서 마주치는 것이, 한 번에 뿌리 끝까지 파고들다 지쳐버리는 것보다 장기 기억 형성에 훨씬 유리합니다. 불안해하지 마세요. 그 '얕은 지식'들이 쌓여 나중에 거대한 지도가 됩니다.

### 4. 다음 타임에 바꿀 것

- 유지할 것과 그 이유

    내 답과 정답이 왜 다른지 추측하고 AI에게 검증받는 과정을 유지할 예정이다.
    내 오답의 한계를 먼저 정의하고 정답의 우월함을 깨닫는 과정이 뇌에 강력한 피드백으로 남는 것 같다.

- 바꿀 것과 그 이유

    문제를 다 푼 후, 가장 흥미로웠던 키워드 하나를 골라 공식 문서의 Deep Dive 섹션이나 기술 블로그를 딱 5분을 정해놓고 읽는 연습을 해야겠다. 지금은 너무 얕게 공부하는 것 같고,그렇다고 너무 input만 만들까봐 불안하니 시간을 정해서 하는 방식을 도입해봐야겠다.
